What is the Native Android Toolkit
Native Android Toolkit is basically a tool that allows your game to access native Android system functions. With that in mind, you can be able to create spectacular games with engaging and fun functions, such as knowing how much time has passed since the last game was closed, this is very useful for building systems for example.
First steps to use
First and foremost is to run DataHandler installation in the first scene of your game. DataHandler is the bridge that enables your app (Unity C#) to talk to the Native Android Toolkit (Java Native of Android). It is important that DataHandler is installed only on the first scene your game plays when it is opened, as DataHandler is indestructible and will be in all subsequent scenes. There are two ways to perform this installation, see below. If DataHandler is not installed on the first scene of your app, most Native Android Toolkit functions and methods simply will not work when your app runs on an Android device.

Running Setup Manually - Setup is a Native Android Toolkit wizard that assists you with the DataHandler installation process. To run it manually, go to: "Tools> MT Assets> Native Android Toolkit> Open Setup".

Setup will assist you in the process of installing the DataHandler.
Using Auto Setup - The dialog below appears whenever a build is run by the Editor, and Native Android Toolkit thinks DataHandler has not been installed in your project. In such cases, Native Android Toolkit will ask you if you would like to install DataHandler.

If you do not wish to install, or have already done the installation, simply click "Do not ask again".
After installation - After installation, a GameObject named "NAT DataHandler" will appear in your scene. This is the DataHandler. If DataHandler is in its initial scene, you don't need to worry about anything else, as everything should be working as expected when you run your app on an Android device.
What is "IonicZip Imported"?
The first time you import Native Android Toolkit into your project, if there is no possible DLL conflicting with IonicZip, NAT will import the DLL called "IonicZip" into your project. This DLL is a DLL that compresses ZIP files and the Native Android Toolkit will use it to unzip the core library (NAT Core.aar) to apply the changes you want and then it will compress this core library back into an AAR file, without you having to edit anything manually! Thanks to this DLL it is possible for you to enable or disable permissions and make other settings all through the "Preferences" window of Native Android Toolkit, through the Editor. Read on to understand how it works!

Resolving compilation errors involving the "Preferences.cs" script
If you have a ZIP file compression/unzipping DLL, or an internal Unity bug occurs, you may experience compilation errors involving the Preferences.cs file. This can happen due to a conflict with the Native Android Toolkit ZIP file compression/unzipping DLL, with some DLL in your project. NAT uses this DLL so that it can apply the changes you make in the "Preferences" window, accessible in "Tools > MT Assets > Native Android Toolkit > Preferences". Although NAT has an internal mechanism to prevent this from happening, you can still experience this problem.

Examples of compilation errors involving Preferences.cs
To correct this problem, it is very simple. Just remove the definition "MTAssets_IonicZip_Available" from your project. To do this, go to Edit> Project Settings> Player. After doing so, your project will recompile and ignore parts of the script, which involve dependencies on the Native Android Toolkit's ZIP library.

If the problem still persists, repeat the step in the paragraph above and delete the script "IonicZip.cs" present in "Asets/MT Assets/Native Android Toolkit/Editor".

As stated earlier, NAT has an internal mechanism to deal with these types of conflicts, but if you have a problem, these steps can help you. In contrast, if you do any of these steps, NAT will not import your compression library, and you will not be able to apply your preferences made in the "Preferences" window. If you have experienced this problem, please contact mtassets@windsoft.xyz for assistance, as there is a possibility that you are experiencing some type of bug. I am always willing to help!

About Preferences
Asset Preferences allow you to customize both the APK that Unity will generate from your app, and the Java side of Native Android Toolkit. Best of all, you don't have to modify a single line of code! To open the preferences window, go to "Tools> MT Assets> Native Android Toolkit> Preferences". Keep reading to understand more about each part.

Demonstration of Preferences window. Please keep in mind that disabling certain permissions will prevent some Native Android Toolkit API methods from working. For example, disabling Bluetooth permission will prevent Bluetooth scanning functions from working. If any Native Android Toolkit function crashes or doesn't work for no apparent reason, make sure the permission attached to it is enabled.
Android Manifest
The Android Manifest is a file called "AndroidManifest.xml" that is in your project. This file is responsible for showing your app's basic information to the Android system upon installation, so the Android system will know everything your app needs, things like permissions, screens, and so on. Native Android Toolkit has tools for managing this file (or creating a new file in your project if the file does not exist). With this tool you can set up basic things in your AndroidManifest.xml without having to edit or understand it! You can for example disable permission requests when your app is first opened, so you have a chance to request permissions yourself whenever you want! AndroidManifest.xml will always be found in the "Assets / Plugins / Android" directory of your project. If this file does not exist, do not worry, your app still has an AndroidManifest.xml, but this will be one automatically generated by Unity with each build. If you want to have your own AndroidManifest.xml, just select the "Yes, Create New If Not Exists" option in Native Android Toolkit preferences, so just save! Once an AndroidManifest.xml exists in your project, you can make changes to it as you like, or you can use Native Android Toolkit for it.

Demonstration of the project AndroidManifest.xml file.
Some editable parameters in AndroidManifest
If you want Native Android Toolkit Preferences to be able to edit your project's AndroidManifest.xml and consequently the APKs generated by your project, you can also change options and preferences! See below which parameters you can change.

Declare Unity Activity - Enables or disables the declaration of the "UnityPlayer" activity (the activity your game runs on) in the manifest. If this activity is not declared, it is as if Unity's activity did not exist.

Is Main Activity - A main activity is the first activity to be performed when the user taps the icon of your application. If Unity activity is not the main activity, it will not be performed when the user taps the icon of your app. This can be useful if you want to configure your manifest so that your app performs another activity when the user taps your app's icon. For example, the activity of a launcher that is displayed before displaying your game activity for example.

Req. Perm. On Open - By default, Unity activity will ask the user for all sensitive permissions (such as camera, files, etc.) right when their application is opened. This can cause a certain amount of user amazement, so if you disable this option, Unity will no longer ask for these permissions as soon as the app opens. This way you will have greater control over the permission request and can use the NAT API to ask the user for these permissions, after explaining what they will be used for!

Notification and detection of manual device time change
The Native Android Toolkit has a small mechanism that monitors the clock of the device on which your application is running, and if this mechanism detects that the user has changed the date or time of your device, manually, the Native Android Toolkit can send you a notification in addition to store this information and the next time your app is opened, your app may have access to that information, to apply some kind of punishment or measure against the possible cheating action performed by the user. This can be useful for games that are based on time to make buildings for example, because if the user advances the time of the device, while his game is closed, he can speed up the construction of things, and that would be a cheat.

Knowing this, you can use some options to configure the alert notification sent to the user when the user changes the device time. You can disable or enable these notifications, in addition to changing the title and text of the notifications. Even if you disable these alerts, NAT will still continue to store information that the time has been changed manually, and that information will be available for your application to access until the next time it is opened. This information is only stored until the next time your app is opened, so if your app is closed, it will only be able to access that information again, if the user changes the time manually, again.

To find out how much time has passed while your app was minimized or closed, and if the user changed the time manually, use the methods "NativeAndroid.Time.GetElapsedTimeLastCloseUntilThisRun()" and the callback "TimeEvents.onResumeAppAfterPause" available in the NAT "Time" class.

Disabling unnecessary permissions
Native Android Toolkit offers many functions, so by default it comes with all the necessary permissions already enabled and included. However, we don't always need all permissions and leaving permissions that our app doesn't use, enabled for the end user to see, may not seem like a good idea. For example, my application may not use file management, so there is no reason for me to request this permission from the user. Knowing this, Native Android Toolkit allows you to disable unnecessary permissions through the Preferences window. But remember that disabling a permission will cause a method that depends on that permission to stop working. For example, disabling the camera permission will cause the methods of turning the deviceâ€™s flash light on and off to stop working. Disabling the file management permission will cause all methods in the file management class to stop working, and so on. You can always refer to each method here in the documentation to find out what permissions each method needs.

Permissions
Wake Up permission and Accurate Notifications
Starting with version M of Android, Google introduced a battery manager in the Android system. This mechanism is capable of putting the Android device in a mode called "deep sleep". This mode means that the device has been idle for a certain time and that is why the operating system turns off several features that do not make sense to remain connected, such as the mobile connection and other services. This saves a lot on the device's battery, however, it reduces the accuracy of some common tasks in some applications, such as the delivery of notifications, since while the device is in deep sleep the operating system manages the delivery of notifications in a that they be rescheduled and grouped with tasks from other applications, tasks that will be executed every 15 minutes for example, in different time windows.

The Android system has a feature called "Wake Lock". This feature allows the application to tell the operating system that such an application needs to remain active for a certain period, taking the device out of its deep sleep, however, depending on the task that such an application performs, battery consumption may increase, even though the application finishes this task, the phone will still take a while to return to its deep sleep, and while it is not in deep sleep, several features and functionality of the system continue to work without limits, until the phone goes back to sleep. When the user unlocks their device, deep sleep is also interrupted.

The problem with deep sleep, as stated earlier, is that it can ruin the way in which scheduled notifications are delivered or other types of tasks are performed. The Native Android Toolkit has notification scheduling functions, which can be affected by the way in which the Twelve of the devices work, or even the battery managers of some Interfaces work, Interfaces such as: MIUI (Xiaomi), ONE UI (Samsung) , EMUI (Huawei) and OxygenOS (OnePlus) for example. These can negatively affect the delivery of notifications not only from the Native Android Toolkit, but also from several Android applications.

Knowing this, the Native Android Toolkit provides the "Accurate Notifications" system which consists of a system that identifies the Android version and applies some checks to try to keep the delivery of notifications as faithfully and accurately as possible, even if Battery Managers get in the way. To activate this function, you only need to leave the "Wake Lock" permission and the "Accurate Notify" option enabled, then the Native Android Toolkit will take care of all inactivity notifications and notifications scheduled by you, trying to ensure that they are all delivered as expected. Keep in mind, "Accurate Notifications" can use up a little more battery power on the user's device.

The way "Accurate Notifications" works is very simple. If the device is not in deep sleep, scheduled notifications will be delivered normally, however, if the device is in deep sleep, NAT will wake you up a few moments before, deliver the notification using Wake Lock and after a few moments the device can go back to sleep. While the device is in deep sleep, it can still delay the delivery of notifications a bit, to ensure that there will be no excessive consumption of battery, this means that sometimes your notifications can take a few minutes longer than expected, to be delivered. The good thing is that with this active system, they will not be killed and simply forgotten by the systems and interfaces of different manufacturers.

Applying preferences after updating NAT
When the Native Android Toolkit is updated from the asset store, a new clean version of the native code core is received in your project, updating the version that existed in your project (NAT Core.aar). All of this native code and resources are stored in the AAR file "NAT Core", including the preferences you save in the Preferences window. After updating your NAT, to ensure that your preferences also apply to the new version of NAT Core, you must apply your preferences again from the Preferences window.

Native Android Toolkit also has a feature in the Editor, which is able to detect whether NAT Core has been updated or modified externally, so NAT will let you know that a NAT Core update has occurred and if you want to apply your existing preferences again to NAT Core, so your preferences will always continue to work!

        {

About Class: Notifications
This class gives you tools to notify your user. With push notifications, you send instant notifications in the status bar of the device in question. With inactivity notifications, your user will receive a notification if they spend 3 days or more without opening your app. With scheduled notifications, you can choose a date/time in the future for a notification to be delivered to the user, and you can schedule up to 20 notifications. These are delivered even if the app is closed and are very useful for notifying the user that something is ready, for example a build. All inactivity or scheduled notifications may be canceled by your app, but only if there are any pending or scheduled notifications in the future.

Schedule a Notification
To schedule a notification to be delivered at a future date, use the code below.

Note: Each scheduled notification depends on a channel to be stored and delivered. There are a maximum of 20 channels in Native Android Toolkit. If you schedule a notification on a channel that already has a notification scheduled, the old one will be overwritten on the channel. If you want to schedule 20 notifications, use a channel for each of the notifications. Make sure you know which channel each notification is scheduled for if you need to cancel a scheduled notification later. A scheduled notification is still stored on a channel and everything is kept even if your app is completely closed. You can cancel a scheduled notification even after restarting your app. Notification will still be delivered even if the app is closed.

Note that: When the device is turned off, it will forget all scheduled notifications, so the user will need to re-open their app for notifications to be scheduled. Delivery of notifications may also be delayed depending on the state of the device. Usually this delay is on average 5 minutes.

Note: Some phones (like Samsung, Xiaomi, Huawei, and others) have battery saver services that can kill the notifications your app schedules. If you notice that your scheduled notifications are not being delivered, add an exception for your app to your device's battery saver service. After that, your scheduled notifications should be delivered normally, but if the problem persists, please contact us for support. This is an unusual problem, but it is helpful to know that this is a possibility!
Script C#


using MTAssets.NativeAndroidToolkit;
using MTAssets.NativeAndroidToolkit.Notify;

public class ExampleScript {

    //Inicialization
	void Start () {
        //Schedule the notification to future (Send notitification in 1 days, 3 hours, 2 minutes on future)
        new NativeAndroid.Notifications.ScheduledNotification(Channel.Ch_1, "Title", "Amazing text")
            .setYearsInFuture(0)
            .setMonthsInFuture(0)
            .setDaysInFuture(1)
            .setHoursInFuture(3)
            .setMinutesInFuture(2)
            .ScheduleThisNotification();
    }
}

About Class: Time
The time class provides tools for you to calculate time, know how much time has passed in certain situations, upload a time from the internet, and so on. Please be aware that time-related Native Android Toolkit methods will always work with the "Calendar" class to return time, or something similar. For example, if you make a request to get the network time, NAT will return you a "Calendar" object that contains the current network date and time. Therefore it is important for you to be aware of and know how the Calendar class works. Keep reading to understand more.
About the "Calendar" class
The Calendar class has as its main goal, to facilitate the manipulation of time, time calculations and so on Unity. This class can be used simply for storing a date / time, calculating sum and subtraction of time, and more. See below for a number of examples of using the Calendar class. You can store time periods within a Calendar object as well as redeem them. For example, you can take the current date, and subtract it from 1 year, then redeem the resulting subtraction values, all quickly!

Note that the Calendar class is a complete class and is sure to help you a lot with time manipulation and calculations!

If you prefer, you can store your app's build times, for example using the Calendar class, and it will be even easier to manipulate your app's build times!

About the "TimeElapsedInfo" class
This is a small Native Android Toolkit class and it has no special tasks. Its only function is to return data about the elapsed time. This class will always be returned with a "Calendar" object (which is the elapsed time, this could be the elapsed time while the app was closed for example), and a boolean variable that tells you if the device date / time was changed manually during this time. period of time. This class is only used by the next two methods you will see below!
Get Elapsed Time Since Last Close of App To This App Open
Native Android Toolkit constantly monitors the way time goes by in your app. Therefore, NAT knows the moment your app was last closed without problems. Keeping this in mind, Native Android Toolkit can calculate how much time has passed while your app has been closed and this is very useful for eg building games where you need to know how much time has passed with the closed game to subtract the construction time! Native Android Toolkit uses the system's common clock to make these calculations and the calculation to know how long it was while the game was closed, is to subtract the date the app was last closed by the exact date it was closed. The app has been opened now. Knowing the user can change this time, Native Android Toolkit also constantly monitors the system clock, and he will know when the user has made any changes to this clock, and will inform you if while the app was closed, the user made a manual change. of date / time. See the code snippet below!

For optimal accuracy, always call the method below in the first scene of your game, at the START method (Start method only), and cache that data. That way your app will know how long exactly it was while it was closed, and you can make your logical calculations normally!
Script C#


using UnityEngine;
using MTAssets.NativeAndroidToolkit;
using MTAssets.NativeAndroidToolkit.Time;

public class ExampleScript
{

    //Inicialization
    void Start()
    {
        //Get elapsed time while the app is closed, this is called when app is started in first scene
        TimeElapsedInfo elapsedTimeInfo = NativeAndroid.Time.GetElapsedTimeLastCloseUntilThisRun();

        //Is date/time changed manually by user, while app is closed?
        Debug.Log(elapsedTimeInfo.dateTimeWasChanged);

        //Time elapsed while app is closed
        Debug.Log(elapsedTimeInfo.elapsedTime);
    }
}

About The "FileRef" Class
The FileRef class allows you to create a reference to an existing file or not in device storage. Once the reference is created and the file/folder exists, you can manage it (read data, rename, copy, move and etc). To create a reference, simply instantiate a new object from the "FileRef" class and enter the path to the desired file or folder. After that, just manage! See below for an example of the FileRef class.
Script C#


using UnityEngine;
using MTAssets.NativeAndroidToolkit;
using MTAssets.NativeAndroidToolkit.Files;
using MTAssets.NativeAndroidToolkit.Time;

public class ExampleScript
{
    /*
     All methods and variables of FileRef class
     */

    //Inicialization
    void Start()
    {
        //Verify if the internal memory is available
        if(NativeAndroid.File.isInternalMemoryAvailable() == false)
        {
            return;
        }

        //Get path of internal memory
        string memory = NativeAndroid.File.GetInternalMemoryPath();

        //Create a reference of a existent file
        FileRef fileRef = new FileRef(memory + "/image.png");

        //If the file not exists, cancel
        if(fileRef.Exists == false)
        {
            return;
        }

        //Delete the file/folder (after delete the file, the fileRef.Exists returns false)
        fileRef.Delete();

        //Get the parent folder of the file/folder
        FileRef parentFolder = fileRef.GetParentFolder();

        //Update the reference of the file/folder (fileRef). This is useful to refresh informations after a file delete, move, verify if exists etc
        fileRef.UpdateReference();

        //Get base informations of this file
        bool isDirectory = fileRef.IsDirectory; //This file of this fileRef is a directory?
        bool isFile = fileRef.IsFile; //This file of this fileRef is a file?
        bool isHidden = fileRef.IsHidden; //This file is hidden?
        bool isWritable = fileRef.IsWritable; //This file is writable?
        Calendar lastModifyDate = fileRef.LastModify; //Date of the last modify of this file

        //If the fileRef is a file. (fileRef.ThisFile allows you to read and write the file of file ref) (only works if the fileRef is a file)
        if (fileRef.IsFile == true)
        {
            //Copy the image.png to the folder "SampleFolder" in root folder of memory. The name of copy is "image.png"
            fileRef.ThisFile.CopyTo(memory + "/SampleFolder");

            //Move the image.png to the folder "SampleFolder" in root folder of memory. After move, original file not exists anymore.
            fileRef.ThisFile.MoveTo(memory + "/SampleFolder");

            //Read all bytes of image.png
            byte[] bytes = fileRef.ThisFile.GetAllBytes();

            //Read the file as string
            string text = fileRef.ThisFile.GetAsTextString();

            //Read the file as image
            Texture2D image = fileRef.ThisFile.GetAsTexture2D();

            //Open this file with the default associated app of system or instaled
            fileRef.ThisFile.Open();

            //Rename the image.png to imageRenamed.png. After this, the fileRef. Exists return false, maybe a new fileRef needs to be created
            fileRef.ThisFile.Rename("imageRenamed.png");

            //Set the image.png to read only
            fileRef.ThisFile.SetReadOnly(true);

            //Get informations of file
            string ext = fileRef.ThisFile.Extension; //Pure extension of this file (returns "png")
            string filePath = fileRef.ThisFile.Path; //The full path for this file
            string pureName = fileRef.ThisFile.PureName; //The pure name of this file (returns "image")
            float sizeInB = fileRef.ThisFile.Size.Bytes; //Returns the size of this file in Bytes
            float sizeInKB = fileRef.ThisFile.Size.Kilobytes; //Returns the size of this file in Kilobytes
            float sizeInMiB = fileRef.ThisFile.Size.Mebibytes; //Returns the size of this file in Mebibytes
            float sizeInMB = fileRef.ThisFile.Size.Megabytes; //Returns the size of this file in Megabytes
            float sizeInGB = fileRef.ThisFile.Size.Gigabytes; //Returns the size of this file in Gigabytes
        }

        //fileRef.ThisFolder allows you to read and write the folder of fileRef. If the fileRef is a file, the fileRef.ThisFolder allows you to read and write the parent folder of the file
        
        //Returns a array of all files present in this folder
        FileRef[] filesInThisFolder = fileRef.ThisFolder.GetAllFiles();

        //Get the name of folder
        string name = fileRef.ThisFolder.Name; //Return the name of folder
        string path = fileRef.ThisFolder.Path; //Return full path of folder
    }
}

Optimization tips for using Native Android Toolkit
Avoid calling functions often in a short time interval (as in Update for example)
Each time you call a NAT function, Unity has to communicate with native Android. If the functions are called excessively, this can cause performance problems, so avoid calling functions in methods such as Update.

Avoid working with gigantic files!
Working with very large files, may cause memory loss on older devices, be careful!
